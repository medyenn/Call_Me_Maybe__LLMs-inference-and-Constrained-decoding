From “I know nothing about AI” to “I can architect a constrained-decoding function-calling engine”

How to use this document

This is not a summary of the subject. It is a curriculum. Each phase builds on the previous one. Do not skip a phase even if it looks “too basic” — the project’s core difficulty (constrained decoding) is 100% built out of the fundamentals in Phases 1–4. Students who jump straight to “how do I mask logits” almost always fail because they never understood why logits need masking in the first place.

Structure of every phase: - Goal — what you should be able to explain/do by the end - Core content — the theory itself - Mental model / analogy — a way to hold the idea in your head - Checkpoint questions — answer these before moving on - Link to the project — how this specific piece of knowledge will resurface later

You asked me not to implement anything. I won’t. Every phase from Phase 8 onward gives you specifications, contracts, and checklists — never code. The actual writing of Python is entirely yours.

PART A — THEORETICAL FOUNDATIONS

PHASE 1 — What actually is a Large Language Model?

Goal

Be able to explain, without hand-waving, what happens when you type a prompt and get text back — at the conceptual level, no math required yet.

Core content

1.1 — A model is a function, not a brain. Strip away the mysticism: an LLM is a mathematical function f(input) → output. The “input” is numbers, the “output” is numbers. Everything else (chat UI, personality, “understanding”) is scaffolding built around this function.

1.2 — Parameters = the function’s dials. A model has billions (or, for your project, ~500 million — “0.6B” is the nominal size class, Qwen3-0.6B) of tunable numbers called weights or parameters. Training is the (very expensive, done once, by someone else) process of tuning these dials so that f(input) tends to produce useful output. You will never train anything in this project — you only ever run (this is called inference) an already-trained model.

1.3 — Training vs. Inference. | | Training | Inference | |—|—|—| | What happens | Weights are adjusted based on data | Weights are frozen, used to compute an output | | Cost | Extremely expensive, done by a lab | Cheap-ish, done by you, on your machine | | What you do in this project | Nothing | Everything |

1.4 — Why “Large”? “Large” refers to parameter count and training data size. Larger models generalize better but are slower and heavier. Your project deliberately uses a small model (0.6B parameters — tiny by industry standards, GPT-class models are 10–1000x bigger) specifically because small models are unreliable at structured output, which is the whole pedagogical point: you’re not compensating with a bigger model, you’re compensating with better decoding engineering.

1.5 — What “generating text” really means. An LLM does not generate a sentence in one shot. It generates one small unit at a time, feeds that unit back into itself, and repeats. This is called autoregressive generation. This single fact is the seed of everything in Phase 3 and Phase 6.

Mental model

Think of the LLM as a very fast, very well-read autocomplete engine that has no memory beyond what’s literally in front of it (the prompt + everything generated so far), and that outputs one puzzle-piece at a time, always asking “given everything so far, what’s statistically the best next piece?”

Checkpoint questions

  • What’s the difference between a parameter and an input?
  • Why does the project deliberately choose a weak model instead of a strong one?
  • True or false: the model “decides” a full sentence before writing it. (False — see Phase 3.)

PHASE 2 — Tokens, Tokenization, and Vocabulary

Goal

Understand what a “token” is, why models don’t operate on letters or words directly, and why this matters enormously for constrained decoding later.

Core content

2.1 — Computers (and models) don’t read text, they read numbers. Before any text reaches the model, it is chopped into chunks called tokens, and each token is mapped to an integer ID. This mapping is called the vocabulary.

2.2 — Tokens are not words. This is the single most important mental correction for beginners. A tokenizer might split:

"What is the sum of 2 and 3?"

into something like:

["What", " is", " the", " sum", " of", " 2", " and", " 3", "?"]

Notice: - Some tokens include a leading space (commonly marked with a special symbol like Ġ in BPE-based tokenizers, or ▁ in SentencePiece-based ones). This lets the tokenizer reconstruct spacing exactly when decoding back to text. - Punctuation is often its own token. - Numbers can be split in surprising ways — "345" might be one token, or might be split into "3" + "45", or "34" + "5", depending on the tokenizer’s training data. This is critical for your project — see the warning box below. - The exact same character sequence can tokenize differently depending on context (e.g., a word at the start of a sentence vs. mid-sentence).

2.3 — The vocabulary file. A tokenizer’s vocabulary is a fixed, finite list — typically tens of thousands of entries — mapping every possible token string to a token ID (and back). This file is not a mystery box: it’s literally a lookup table you can open and read. The llm_sdk SDK gives you get_path_to_vocab_file() for exactly this reason.

2.4 — Encoding and decoding. - Encode: text → list of token IDs (what you feed the model) - Decode: list of token IDs → text (what you show a human)

2.5 — Why tokenization is the crux of constrained decoding. Constrained decoding works by deciding, at each generation step, which token IDs are currently legal. To do that, you must be able to answer: “if I appended this candidate token’s string to what I’ve generated so far, would the result still be progressing toward valid JSON that matches my schema?” That means you need to reason about the string representation of every token in the vocabulary, not just abstract IDs. This is why understanding the vocabulary file (Phase 2.3) is non-negotiable before touching Phase 6.

⚠ Important trap to understand now (not later)

Because numbers can be split across multiple tokens in non-obvious ways, and because “quote”, “colon”, “comma”, “brace” might each be their own token or might be fused with neighboring characters, you cannot assume “one token = one JSON character” or “one token = one digit.” Your constrained decoder has to work at the string level, testing each candidate token by simulating what the text would look like if that token were appended — not by assuming a clean 1:1 correspondence with JSON syntax.

Mental model

Think of the vocabulary as a giant multiple-choice answer sheet. At every step, the model doesn’t “type a letter” — it picks one bubble out of ~30,000–150,000 pre-printed possible chunks of text. Your job in constrained decoding is to cover up the bubbles that would produce garbage, before the model gets to pick.

Checkpoint questions

  • Why can’t you assume a digit always corresponds to exactly one token?
  • What two SDK methods will you need to inspect the vocabulary and its mapping to IDs?
  • If token " the" (with leading space) exists, what does the leading-space marker tell you about reconstructing the original text?

PHASE 3 — Logits, Probability, and Token Selection

Goal

Understand what a logit is, how a token gets chosen, and how you can intervene in that choice — which is the literal mechanism of constrained decoding.

Core content

3.1 — What is a logit? After processing the input token IDs, the model doesn’t directly output “the next token.” It outputs one raw score (a “logit”) for every single entry in the vocabulary. If the vocabulary has 150,000 entries, the model produces 150,000 numbers, one per possible next token. Higher = the model thinks this token is more likely to come next.

3.2 — From logits to probabilities: softmax. Logits are raw, unbounded numbers (can be negative, can be huge). To turn them into a proper probability distribution (all values between 0 and 1, summing to 1), you apply the softmax function. You don’t need to derive softmax by hand for this project, but you must understand conceptually: softmax preserves relative ordering — the highest logit becomes the highest probability. This matters because of 3.4 below.

3.3 — Selection strategies (how the “winning” token is picked). - Greedy decoding: always pick the single highest-probability token. Deterministic, can be repetitive. - Sampling (temperature, top-k, top-p / nucleus sampling): introduce controlled randomness by sampling from the (adjusted) distribution instead of always taking the max. Used for creative, varied text. - For this project, once you apply constraints (Phase 6), you will typically want deterministic or near-deterministic selection among the valid tokens — creativity is not the goal, correctness is.

3.4 — The critical trick: you can edit logits before selection. This is the single mechanical fact that makes the entire project possible:

Nothing stops you from taking the raw logits array, and setting the logit of any token you consider “invalid” to negative infinity (or some very large negative number), before applying softmax/selection.

Once a logit is -inf, softmax will assign that token a probability of essentially zero. It becomes mathematically impossible for the decoding process to pick it — not “unlikely,” impossible. This is not a hack; it is the standard, industry-recognized technique called constrained decoding or guided generation, and it’s exactly what production systems (OpenAI function calling, Anthropic tool use, open-source libraries like outlines or guidance) do internally — except you are forbidden from using those libraries (§IV.3.1) precisely so you build the mechanism yourself and truly understand it.

3.5 — The generation loop, precisely. Subject §V.3.2 gives you this loop; here it is annotated:

1. Prompt (text)
2. Tokenize → list of tokens (strings)
3. Encode → list of token IDs (integers)
4. Feed IDs into the model → get_logits_from_input_ids()
5. Get logits (one score per vocab entry)
   ↳ [YOUR INTERVENTION HAPPENS HERE — Phase 6]
6. Select next token ID (argmax, or sampling, over the surviving/valid logits)
7. Append the new token ID to the sequence
8. Repeat from step 4, using the *extended* sequence as input
   ... until a stop condition is met (e.g., you've completed valid JSON,
       or hit a max length, or generated an end-of-sequence token)

Every single token in your output is produced by running this entire loop again — the model has no “memory” of having thought about the whole sentence in advance; it is re-deriving its next-token guess from scratch (well, using cached computation internally, but conceptually) every single step.

Mental model

Imagine a game show where a contestant (the model) is shown a partial sentence and must guess the next word from a giant list of flashcards (the vocabulary), each with a “confidence score” (logit) pre-assigned. You, the game show host, are allowed to physically remove flashcards from the table before the contestant picks — the contestant can only choose from what remains. That’s constrained decoding. The contestant never knows cards were removed; it just picks the best of what’s left.

Checkpoint questions

  • What is the difference between a logit and a probability?
  • Why does setting a logit to -infinity guarantee a token cannot be chosen (rather than just making it “less likely”)?
  • Why is greedy decoding usually preferred over creative sampling for this specific project’s goal?

PHASE 4 — Function Calling as a Concept

Goal

Understand what “function calling” means at the LLM level, why it’s not the same as “the LLM does math,” and what problem it actually solves.

Core content

4.1 — The core insight (subject §III.1). An LLM asked “What is the sum of 40 and 2?” does not compute — it pattern-matches against text it has seen and, if you’re lucky, produces "42" as plausible-looking text. It has no calculator, no database access, no real “execution” capability by itself. Function calling doesn’t ask the LLM to solve the problem. It asks the LLM to translate a natural-language intent into a precise, executable instruction — a function name plus correctly-typed arguments — which something else (your program, a real calculator function, an API, a database) will then actually execute.

4.2 — The division of labor.

Natural language  →  [LLM: intent recognition + argument extraction]  →  Structured call
Structured call   →  [Your program / a real function]                  →  Actual execution

The LLM never touches the real answer. It only ever decides which tool and with what typed inputs. This is why your project’s output format (§V.4) is exactly {prompt, name, parameters} and never a computed result.

4.3 — Why this generalizes far beyond calculators. Subject §III.2 lists real uses: calling APIs, querying databases, controlling devices, chaining multi-step tasks, and — importantly for you to understand the “why” — extracting structured data from unstructured text (e.g., pulling {protagonist_name, protagonist_sex, protagonist_age} out of a novel). Function calling and “structured extraction” are the same underlying mechanism: forcing free text into a fixed schema.

4.4 — The reliability problem (subject §III.3), restated precisely. If you ask nicely (“please respond only in JSON matching this schema”), a small model will frequently: - add prose before/after the JSON (“Sure! Here’s the JSON: {…}”) - produce syntactically broken JSON (missing braces, trailing commas) - invent a function name that doesn’t exist - put a string where a number was required - omit required fields

None of this is a “bug” you can prompt your way out of reliably at 0.6B parameters. Subject explicitly forbids this approach: “Your solution must NOT rely on the model spontaneously producing correct JSON from a prompt.” The fix isn’t a better prompt — it’s controlling the generation mechanism itself, which is exactly Phase 3.4’s technique, applied with a JSON grammar and a function schema. That’s Phase 5 and 6.

4.5 — Two separate sub-problems inside “function calling.” It helps to mentally split the task into two independently-reasoned-about problems: 1. Function selection: given N available functions and a prompt, which one is the right one? (A classification-like decision the LLM must make.) 2. Argument extraction & typing: given the chosen function’s schema, pull the correct values out of the prompt and coerce them to the correct type (number vs. string vs. boolean, etc.)

Both sub-problems need the LLM’s language understanding. Both sub-problems need constrained decoding to guarantee the output format is trustworthy regardless of whether the content is correct.

Mental model

Function calling turns the LLM into a very well-read but very literal receptionist: it doesn’t perform surgery, cook your meal, or do your taxes — it listens to what you want and correctly fills out the right request form, with the right fields, in the right format, and hands it to someone (or something) that actually does the work.

Checkpoint questions

  • Why does the LLM never produce 42 as the final answer in this project?
  • Name the two sub-problems bundled inside “function calling.”
  • Why is “just prompt it better” explicitly rejected as a solution strategy?

PHASE 5 — Structured Output, Schemas, and JSON as a Grammar

Goal

Understand JSON not as “a text format” but as a formal grammar — a strict set of rules about what character can legally follow what — because that reframing is what makes constrained decoding tractable.

Core content

5.1 — JSON has a grammar, just like a programming language. JSON’s full grammar is officially specified (see RFC 8259). You don’t need to memorize it, but you need the mental shift: at any point while writing valid JSON, only a finite, well-defined set of next characters is legal. For example: - Right after {, legal next non-whitespace characters are: " (start of a key) or } (empty object). - Right after a closing " of a key, the only legal next non-whitespace character is :. - Right after :, legal next characters depend on the expected value type: " for a string, a digit or - for a number, t/f for true/false, [ for an array, { for a nested object, n for null. - Inside a string, almost anything is legal except an unescaped " (which closes the string) or unescaped control characters.

This “at every position, only some next characters are legal” property is exactly what a finite state machine (FSM) or pushdown automaton models. You do not need to build a formal FSM library — but you should recognize that your “am I still on track for valid JSON?” logic is essentially you hand-rolling a state tracker.

5.2 — Structure validity vs. schema validity — two different constraints, both required. The subject is explicit about this distinction (§V.3.3): your decoder must enforce both: 1. Syntactic JSON validity — braces balanced, commas in the right places, strings properly quoted, etc. (This alone would let a model produce {"name": "banana", "parameters": {"x": "yes"}} — syntactically perfect, but useless.) 2. Schema compliance — the object must specifically match one of the functions defined in functions_definition.json: the right key names, and each value must match its declared type (number, string, boolean, etc.) — not just “any valid JSON value.”

5.3 — What a “schema” is, concretely, for this project. Look again at functions_definition.json (§V.2): each function is described by a name, a description (for the LLM to read to choose the right function — this text is input to your function-selection prompt), a parameters object (each parameter name mapped to a type), and a returns type (irrelevant to your output, but part of the definition). This is conceptually identical to JSON Schema — a machine-readable description of what a valid JSON object must look like. You are, in effect, building a mini JSON-Schema-constrained JSON generator from scratch.

5.4 — Type coercion. “Number” in JSON doesn’t distinguish int vs. float — but your subject’s example output (§V.4.1) shows "a": 2.0, "b": 3.0 — floats, even though the prompt said whole numbers 2 and 3. This tells you the target representation for "number" in this project is a JSON floating-point-compatible token sequence (digits, optional ., optional exponent) — not necessarily “must always contain a decimal point,” but your grammar for the number type must be broad enough to accept how the model naturally continues digit sequences, while never producing something invalid like "1.2.3" or "1,000".

Mental model

Think of writing valid+schema-compliant JSON like filling out a government form with a strict grid layout: at box #4 you are only allowed to write a number, at box #7 you’re only allowed a checkbox mark. A constrained decoder is a strict clerk standing over the applicant’s shoulder, physically blocking the pen from touching any part of the form except the currently-legal region, and only allowing legal-for-that-field symbols even within the region.

Checkpoint questions

  • What’s the difference between “syntactically valid JSON” and “schema-compliant JSON”? Give an example of a string that is the former but not the latter.
  • Why is thinking of JSON as a “grammar with states” more useful than thinking of it as “just a text format” when you’re about to build a constrained decoder?
  • Where in your project does the description field of a function definition get used, and why is it different in purpose from the parameters field?

PHASE 6 — Constrained Decoding, End to End

Goal

Assemble Phases 1–5 into the complete conceptual algorithm you’ll eventually implement. This is the theoretical heart of the mandatory part. By the end of this phase you should be able to narrate, step by step, in plain English, exactly what your program will do — without writing a line of code.

Core content

6.1 — The algorithm, narrated.

At a high level, generation of one JSON output proceeds like this:

  1. Establish the target schema for the current step. Before you generate a single token, you must know: “am I currently expected to write a {? a key name? a colon? a value of type X? a comma-or-closing-brace?” This running notion of “where am I in the grammar, and what’s legal next” is your decoding state.
  2. Ask the model for logits given everything generated so far (get_logits_from_input_ids).
  3. Build the set of allowed next tokens, given the current decoding state. This is the hard part, and it decomposes into two checks per candidate token:
    • Structural check: if I append this token’s string to the output-so-far, does the result remain a valid prefix of some completable JSON string? (Not necessarily complete yet — but not yet broken either.)
    • Schema check: does this token’s string keep me within the value-space allowed for the current field (e.g., only digits/./- while inside a number value; only characters that keep a string value literal; only the exact remaining characters of a legal key name when I’m about to write a key)? Practically, this means: for every token ID in the vocabulary, decode it to its string form, simulate appending it, and test both checks. (You will want to be smart about not brute-forcing all ~100k+ tokens naively every single step for performance — see §V.5 and Phase 9’s performance notes — but conceptually, this is the check being performed.)
  4. Mask the logits: for every token that fails either check, set its logit to negative infinity (Phase 3.4).
  5. Select the next token from the surviving logits (typically argmax/greedy, since you want reliability, not creative variety).
  6. Append the chosen token, advance your decoding state (e.g., if you just closed a string value that matched a known-required key, you know a comma-or-brace comes next).
  7. Repeat until the JSON object is structurally and semantically complete (you’ve closed the outermost } and satisfied all required keys).

6.2 — Function selection happens before or as part of this loop. Subject: “The function to call should be chosen using the LLM, not with heuristics or any other sort of medieval magic.” This means: you cannot simply keyword-match the prompt against function names/descriptions with string .contains() logic. The LLM itself, through its logits, must be the thing deciding which function name gets written into the "name" field — meaning your constrained decoder’s allowed-token-set at that position must be scoped to “one of the valid function name strings, verbatim,” and letting the model’s own logits pick which one, among only those legal options. (This is your Phase 3.4 mechanism used for a closed set of allowed literal strings — a simpler case than the general “number” or free-text “string” constraint.)

6.3 — Why this guarantees 100% valid, parseable output. Because at every single token boundary, illegal continuations are mathematically impossible to select (masked to -inf), the model is structurally incapable of producing anything except a well-formed, schema-compliant JSON object. It might still choose the wrong function or the wrong number (that’s an accuracy problem, targeted at 90%+ per §V.5) — but it can never produce a result that fails to parse, or that violates the schema’s structure. This is the precise meaning of subject’s claim: “the produced JSON is 100% retrievable and can always be parsed without errors.” Accuracy (semantic correctness) and validity (structural correctness) are two different guarantees, achieved by two different means: validity by masking, accuracy by the model’s own (imperfect) language understanding.

6.4 — Where creativity/randomness has no place here. Unlike chatbots or story generation, you generally want deterministic, greedy selection among the legal tokens. There is no “more interesting” JSON — there is only “correct” and “incorrect,” so temperature/top-p sampling mostly just adds noise you don’t want, though you’re free to experiment.

Mental model

You are not writing a program that “asks the AI to write JSON.” You are writing a guided autocomplete keyboard that happens to have an AI brain plugged into it for content decisions (which function? which number? which name?), while you retain full physical control over which keys are even pressable at each moment.

Checkpoint questions

  • Walk through, in words, what happens at the exact token position right after the model has written {"name": "fn_add_numbers", "parameters": {"a": 2.0,. What set of next tokens should be legal, and why?
  • Why does masking illegal logits to -inf guarantee validity, while nothing guarantees accuracy?
  • Why is “choose the function via keyword matching” explicitly against the spirit (and the letter) of the subject?

10.2 — Component-by-component specification

For each component below: Responsibility, Inputs, Outputs, Must handle, Must NOT do.

A. CLI / Entrypoint (src/__main__.py or equivalent module entry)

  • Responsibility: parse --functions_definition, --input, --output; apply documented defaults; hand off to the orchestrator; catch any exception that bubbles all the way up and convert it into a clean, human-readable error message + non-zero exit code (never a raw traceback dump to the end user).
  • Inputs: sys.argv.
  • Outputs: process exit code; delegates real work to the Orchestrator.
  • Must handle: missing arguments (use defaults), --help (standard argparse behavior is fine), any unhandled downstream exception (top-level try/except around the pipeline invocation).
  • Must NOT do: contain any business logic itself (no JSON parsing, no model logic) — it is purely a thin wiring layer.

B. Config / Defaults component

  • Responsibility: define the three default paths (§IV.3.2 / §V.4) in exactly one place so they’re never duplicated/typo’d across the codebase.
  • Inputs: none (or CLI overrides passed through).
  • Outputs: resolved Path objects for functions-definition file, input file, output file.
  • Must handle: relative vs. absolute paths cleanly, directory creation for the output path if the parent directory doesn’t yet exist (still don’t crash if it’s missing — create it, or fail with a clear message if creation isn’t possible).

C. Function Definitions Loader + Data Model

  • Responsibility: read functions_definition.json, parse and validate it into typed pydantic models (this is where §IV.3.1’s “all classes must use pydantic” requirement lands most naturally): a FunctionParameter model (name → type), a FunctionDefinition model (name, description, parameters, returns), and a top-level FunctionCatalog wrapping the list.
  • Inputs: file path.
  • Outputs: a validated, typed in-memory catalog of functions.
  • Must handle: file not found, invalid JSON syntax, JSON that doesn’t match the expected schema (e.g., missing name key) — each with a distinct, clear error message; an empty array of functions (valid JSON, zero functions — decide and document what your program does here, e.g. clean failure with a clear message, since no function calling is possible).
  • Must NOT do: assume specific function names or a specific number of functions anywhere in its logic (§V.4.2 warning).

D. Prompts Loader + Data Model

  • Responsibility: read function_calling_tests.json into a list of typed PromptRequest pydantic models (prompt: str).
  • Inputs: file path.
  • Outputs: list of validated prompt objects.
  • Must handle: same file-error classes as component C; an empty prompts array (valid → produce an empty output array, not an error); a prompt entry missing the "prompt" key or with a non-string value.

E. Vocabulary Manager

  • Responsibility: load the vocabulary file (via llm_sdk’s get_path_to_vocab_file()), and expose a clean bidirectional mapping: token_id → token_string and (if useful) token_string → token_id. This is the component every part of the decoding engine will query when it needs to know “what string would token ID N actually produce if selected?”
  • Inputs: path returned by the SDK.
  • Outputs: an in-memory lookup structure (built once, reused for every generation step and every prompt — see performance notes in 10.3).
  • Must handle: special marker characters for leading spaces / sentencepiece markers (Phase 2.2) — your mapping must expose the actual resulting text, not the raw marked-up token string, when a component asks “what would this produce?”
  • Must NOT do: silently assume ASCII-only tokens; vocabularies can include multi-byte / unicode entries.

F. Constrained Decoding Engine — the heart of the project

This deserves to be split conceptually into sub-components, even if you organize them as one module later:

F.1 — Grammar / State Tracker - Responsibility: maintain “where am I in the JSON structure right now” (e.g., EXPECTING_OPEN_BRACE, EXPECTING_KEY_STRING, EXPECTING_COLON, EXPECTING_VALUE_OF_TYPE_X, EXPECTING_COMMA_OR_CLOSE, DONE), and expose, at any moment, a method like “given the current state and a candidate next-token string, would this be a legal continuation?” - Inputs: current state + generated-so-far text; a candidate token string. - Outputs: boolean legality + the new state that would result if accepted. - Must handle: nested contexts correctly (you’re inside the parameters object, which is itself inside the top-level object) — this naturally wants a small stack-like structure, since JSON nesting is exactly a pushdown-automaton problem (Phase 5.1).

F.2 — Schema Constraint Resolver - Responsibility: given the currently expected field (which key of which function we’re filling), consult the FunctionDefinition model (component C) to determine the expected type, and provide type-specific legality checks (e.g., “for a number value in progress, is this candidate token consistent with continuing a valid JSON number literal?”; “for a string value, are there no unescaped bare quotes/control characters?”; “for the top-level name field, is this candidate token consistent with being a prefix of at least one function’s real name?”). - Inputs: current field context, FunctionCatalog. - Outputs: a per-field legality predicate. - Must handle: the function name field specifically needs a “closed-set of literal strings” checker — different in kind from the “number” and “string” checkers, which are open-ended but type-constrained, not closed-set.

F.3 — Logit Masker - Responsibility: given the full logits vector and the current legal-token-ID set (derived by combining F.1 + F.2 + the Vocabulary Manager), produce a masked logits vector where illegal entries are set to negative infinity. - Inputs: raw logits (List[float]), legal token ID set. - Outputs: masked logits. - Must NOT do: mutate the original logits list in a way that surprises the caller — treat this as a pure function returning a new structure, for testability.

F.4 — Token Selector - Responsibility: given masked logits, pick the next token ID (greedy argmax by default, per Phase 6.4’s reasoning). - Inputs: masked logits. - Outputs: single chosen token ID. - Must handle: the degenerate case where no tokens are legal (should not happen if F.1/F.2 are correct, but defensively: this indicates a bug in your grammar tracker, and should raise/log a clear internal error rather than crash opaquely or silently produce garbage).

F.5 — Generation Loop Driver - Responsibility: orchestrate F.1–F.4 across the full multi-step loop described in Phase 6.1/6.3, calling get_logits_from_input_ids each iteration, terminating once the state tracker reports DONE, and returning the final generated text (which is then trivially json.loads-able, by construction). - Inputs: prompt text (encoded via SDK encode), function catalog. - Outputs: raw generated JSON text. - Must handle: a maximum-iteration safety cap (even though structurally you should always terminate, a hard iteration ceiling protects against an unforeseen grammar bug turning into an infinite loop — respects §IV.1’s “never crashes / never hangs” spirit even though it’s phrased as “never crashes”).

G. Output Assembler & Validator

  • Responsibility: take the raw generated JSON text per prompt, parse it (json.loads — trivial by this point, since it’s guaranteed valid), wrap it into a pydantic FunctionCallResult model (prompt, name, parameters), and run a final defensive validation pass against the actual FunctionDefinition (belt and suspenders: even though your grammar should make this redundant, validating the final result before writing catches any lingering engine bugs before they reach the deliverable file).
  • Inputs: generated text, original prompt, function catalog.
  • Outputs: a validated FunctionCallResult.
  • Must handle: report a validation failure clearly rather than writing malformed data — decide and document your policy for what happens to the output array if a single prompt fails (skip it with a warning? include a best-effort partial? This is a design decision to document in your README per §VI).

H. Output Writer

  • Responsibility: serialize the full list of FunctionCallResult objects to the target output path as a pretty or compact JSON array; ensure parent directories exist; use a context manager for the file handle.
  • Inputs: list of results, output path.
  • Outputs: a file on disk.
  • Must handle: permission errors, disk errors — clear message, no crash.

I. Orchestrator / Pipeline Controller

  • Responsibility: the only component that “knows about everything” — wires Config → Loaders → Vocabulary Manager → (per prompt) Decoding Engine → Output Assembler → Output Writer, and reports overall progress/timing (relevant to the 5-minute performance target in §V.5).
  • Inputs: resolved config paths.
  • Outputs: none directly (delegates to Output Writer); return code / summary for the CLI layer.

J. Error Handling & Logging (cross-cutting, not a “box” but a discipline)

  • Define your own small hierarchy of custom exceptions (e.g., a base project-specific error type, with subtypes for “input file error,” “schema validation error,” “generation error”) so that the top-level CLI handler (component A) can catch broadly but still report specifically what went wrong.
  • Every component above should raise your custom exceptions rather than letting raw KeyError/json.JSONDecodeError/etc. propagate unexplained — catch low-level exceptions close to their source and re-raise as a clear, contextualized project exception (“Failed to parse functions_definition.json: ” rather than a bare JSONDecodeError traceback).

10.3 — Performance considerations (so §V.5’s 5-minute budget isn’t a surprise

at the end) - Recomputing “for every one of possibly 100k+ vocabulary entries, decode the token and run the grammar/schema check” at every single generation step, for every prompt, naively, is the most likely source of a performance failure. Plan, from the start, for: - Precomputing the token_id → token_string table once (Vocabulary Manager is built once, not per prompt). - Where possible, precomputing coarse categories of tokens once (e.g., “the set of tokens that are pure ASCII digits,” “the set of tokens that are exactly one of the function names’ first N characters”) rather than re-deriving them from scratch every step. - Keeping the “legal token set” computation as narrow as possible given the current grammar state, rather than scanning the entire vocabulary unfiltered at every step when a cheaper structural shortcut exists (e.g., once you’re definitely inside a closed-set field like the function name, you only need to consider tokens that are plausible prefixes/continuations of the small number of real function names — not the whole vocabulary). - This is a natural place for the “bonus: performance optimizations (caching, batching)” item, but a baseline level of this thinking is needed even for the mandatory 5-minute target once your function/prompt counts grow.

Checkpoint questions

  • Which component is responsible for guaranteeing validity, and which is responsible (as best as an LLM can be) for accuracy? Why must these stay separated?
  • If functions_definition.json is updated by a reviewer to include a brand-new function with a brand-new parameter type keyword you’ve never seen, which components should need to change, and which should require zero changes? (A good architecture should make the honest answer “almost nothing needs to change, except perhaps adding support for one new type-keyword in the Schema Constraint Resolver.”)
  • Why does the Output Assembler still validate the result even though the decoding engine is supposed to already guarantee validity?

Model Fields (name: type) Validation notes
ParameterSpec type: str Must be one of your supported type keywords (number, string, boolean, and any others you decide to support — document unsupported-type behavior)
FunctionDefinition name: str, description: str, parameters: dict[str, ParameterSpec], returns: ParameterSpec name should be non-empty; consider a light validator ensuring uniqueness of name across the catalog

PHASE 12 — Task Breakdown & Suggested Build Order

Goal

A concrete, ordered checklist of implementation tasks — the sequence you should actually code in, so you always have something runnable and testable at each step, rather than a mountain of code that only “works” at the very end.

Reminder: you implement each task. This is the outline of what and in what order, never how (no code).

  1. Scaffolding. Create src/, pyproject.toml, .gitignore, empty Makefile targets that at least run without error (install, run, debug, clean, lint), copy llm_sdk/ in.
  2. CLI + Config. Argument parsing with defaults; verify uv run python -m src --help works and prints something sane; no real logic yet.
  3. Data models. Write the pydantic models from Phase 11, with no loaders yet — just the shapes, verified with a few hand-written throwaway examples.
  4. Loaders. Implement Function Definitions Loader and Prompts Loader; test against the two example files in the subject, and against deliberately broken files (missing, malformed) to confirm clean error messages.
  5. Vocabulary Manager. Load the vocab file via the SDK; write a tiny sanity check that a handful of token IDs decode to the strings you expect.
  6. Grammar/State Tracker (F.1) in isolation. Before touching the model at all, hand-test your state tracker against hand-written candidate strings (e.g., “is { legal in the initial state? is } legal right after {? is , legal right after a closed string value?”) — this component can and should be fully verified without ever calling the LLM.
  7. Schema Constraint Resolver (F.2) in isolation. Same idea — test against a hand-built tiny fake FunctionCatalog, no model involved yet.
  8. Wire F.1+F.2 into the Logit Masker + Token Selector (F.3+F.4), with a fake logits source (e.g., a stub returning random logits) just to prove the masking mechanics work end-to-end without waiting on real model inference.
  9. Integrate the real LLM SDK into the Generation Loop Driver (F.5). Run against a single trivial prompt (e.g., “What is the sum of 2 and 3?”) and manually inspect the raw generated text.
  10. Output Assembler + Writer. Wire the loop’s output into the final file format; run the full pipeline end-to-end on the subject’s example files.
  11. Batch over all prompts + timing. Confirm the full example test set completes well within 5 minutes; start profiling/optimizing only if needed.
  12. Edge case hardening. Deliberately feed: empty prompt list, missing files, malformed JSON, an ambiguous prompt, a prompt requiring a function with multiple parameters, large numbers, special characters in strings.
  13. Lint/type-check pass. Run make lint (and lint-strict if attempting it) until clean.
  14. Tests (not graded, but strongly recommended per §IV.3). Write pytest coverage for the isolated components (steps 6, 7) at minimum — these are the cheapest, highest-value tests since they don’t need the model.
  15. README. Write it last, once every design decision is actually final, so it accurately documents what you built rather than what you planned to build.
  16. Bonus features, only after 1–15 are solid and documented.
